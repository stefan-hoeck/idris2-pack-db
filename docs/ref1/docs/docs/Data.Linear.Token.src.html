 24 | ||| A list of arbitrary managed resources.
 25 | |||
 26 | ||| This is used as parameter for the linear token `T1`
 27 | ||| provided by this library. The idea is that only resources
 28 | ||| bound to the linear token can be used in the current linear
 29 | ||| context.
 30 | |||
 31 | ||| We distinguish between two cases: Values that belong to `IO` land
 32 | ||| (i.e. mutable references and arrays coming from and being sent
 33 | ||| to the FFI), and values that should be created (and, possibly, release)
 34 | ||| in pure computations.
 35 | |||
 36 | ||| For the former types of values, `Resources` should be equal to `[World]`,
 37 | ||| and the corresponding types should implement interface `InIO`. These
 38 | ||| values can be freely used in linear computations involving a linear
 39 | ||| token of type `T1 [World] a`, but running these computations is an `IO`
 40 | ||| action.
 41 | |||
 42 | ||| For the latter type of values (those being used in pure computations),
 43 | ||| `T1 rs a` should be used, where `rs` is a list of linear resources bound
 44 | ||| to the linear token.

 68 | ||| A linear token used in computations with mutable linear state.
 69 | |||
 70 | ||| The idea is that mutable state (mutable references, arrays, pointers,
 71 | ||| and so on) can only be accessed in the presence of the linear token
 72 | ||| that was used to create the mutable resource.
 73 | |||
 74 | ||| The advantage of using linear types over the `ST` monad: They come
 75 | ||| without the (significant) performance overhead of monadic code in
 76 | ||| Idris. In addition, this provides safe resource handling, because
 77 | ||| the function for running linear computations over a `T1` (function
 78 | ||| `run1` must work with function argument `f` of type
 79 | ||| `(1 t : T1 []) -> R1 [] a`, that is, any resources required must
 80 | ||| be allocated and released within `f`.

 83 | T1 rs = %World
