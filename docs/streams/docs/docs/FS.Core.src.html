 26 | --------------------------------------------------------------------------------
 27 | -- Pull Type
 28 | --------------------------------------------------------------------------------
 29 |
 30 | ||| A `Pull f o es r` is a - possibly infinite - series of effectful
 31 | ||| computations running in monad `f`, producing an arbitrary sequence
 32 | ||| of chunks of values of type `c o` along the way, that eventually
 33 | ||| terminates either with an error of type `HSum es` or with a result
 34 | ||| of type `r`.
 35 | |||
 36 | ||| A `Pull` is a monad with relation to `r`, so a sequencing of `Pull`s
 37 | ||| via bind `(>>=)` means a sequencing of the chunks of output generated
 38 | ||| along the way. For instance, the following `Pull` will produce the
 39 | ||| values `[1,2,3,4,5,6]` when being run:
 40 | |||
 41 | ||| ```idris2
 42 | ||| emit [1,2,3] >> emit [4,5,6]
 43 | ||| ```
 44 | |||
 45 | ||| A `Pull` provides capabilities for safe resource management via
 46 | ||| `FS.Scope`s, allowing us to release allocated resources as soon as
 47 | |||
 48 | |||  * the pull is exhausted and produces no more output.
 49 | |||  * evaluating the pull aborts with an error.
 50 | |||  * a pull will not longer be asked to produce additional values
 51 | |||    because an outer pull has been exhaused.
 52 | |||
 53 | ||| The last case occurs for instance when we stop evaluating a `Stream`
 54 | ||| via `take` or `takeWhile`.
 55 | |||
 56 | ||| About the effect type `f`: Most non-trivial `Pull`s and `Stream`s come
 57 | ||| with the potential of failure. In addition, running a `Pull` to completion
 58 | ||| requires (thread-local) mutable state to keep tracks of the evaluation
 59 | ||| scopes. Effect type `f` must therefore implement interface `Data.Linear.ELift1`
 60 | ||| at the very least. This still allows us to run a `Pull` or `Stream`
 61 | ||| as a pure computation, for instance when using `Control.Monad.Elin`.
 62 | |||
 63 | ||| Most of the time, however, we are interested in streams that produce
 64 | ||| arbitrary side-effects, and an I/O monad such as `IO.Async` is required.
 65 | |||
 66 | ||| A note on totality: Theoretically, we should be able define `Bind` as follows:
 67 | |||
 68 | ||| ```idris
 69 | ||| Bind : Pull f o es r -> Inf (r -> Pull f o es s) -> Pull f o es s
 70 | ||| ```
 71 | |||
 72 | ||| This would allow us to safely generate total, infinite streams of output
 73 | ||| and only running a `Pull` would be a non-total function. In practice, however,
 74 | ||| I found this approach to have various shortcomings: We'd need a custom
 75 | ||| bind operator for defining infinite pulls. That's not hard to implement, but
 76 | ||| I found it to be fairly limited and cumbersome to use, especially since
 77 | ||| type inference got very erratic. For the time being, I therefore decided
 78 | ||| to resort to `assert_total` when defining potentially infinite streams.
 79 | ||| It's not pretty, but it gets the job done.
