 12 | interface MErr f => MCancel f where
 13 |   ||| Requests self-cancelation of the current fiber (computational thread).
 14 |   canceled : f es ()
 15 |
 16 |   ||| Masks cancelation on the current fiber. The argument to `body` of type `Poll f` is a
 17 |   ||| natural transformation `f ~> f` that enables polling. Polling causes a fiber to unmask
 18 |   ||| within a masked region so that cancelation can be observed again.
 19 |   ||| 
 20 |   ||| In the following example, cancelation can be observed only within `fb` and nowhere else:
 21 |   ||| 
 22 |   ||| ```
 23 |   |||   uncancelable $ \poll => fa >> poll(fb) >> fc
 24 |   ||| ```
 25 |   ||| 
 26 |   ||| If a fiber is canceled while it is masked, the cancelation is suppressed for as long as the
 27 |   ||| fiber remains masked. Whenever the fiber is completely unmasked again, the cancelation will
 28 |   ||| be respected.
 29 |   ||| 
 30 |   ||| Masks can also be stacked or nested within each other. If multiple masks are active, all
 31 |   ||| masks must be undone so that cancelation can be observed. In order to completely unmask
 32 |   ||| within a multi-masked region the poll corresponding to each mask must be applied to the
 33 |   ||| effect, outermost-first.
 34 |   ||| 
 35 |   ||| ```
 36 |   |||   uncancelable $ \p1 =>
 37 |   |||     uncancelable $ \p2 =>
 38 |   |||       fa >> p2(p1(fb)) >> fc
 39 |   ||| ```
 40 |   ||| 
 41 |   ||| The following operations are no-ops:
 42 |   ||| 
 43 |   |||   1. Polling in the wrong order
 44 |   |||   2. Subsequent polls when applying the same poll more than once: `poll(poll(fa))` is
 45 |   |||      equivalent to `poll(fa)`
 46 |   |||   3. Applying a poll bound to one fiber within another fiber
 47 |   uncancelable : (body: Poll f -> f es a) -> f es a
 48 |
 49 |   ||| Registers a finalizer that is invoked if cancelation is observed during the evaluation of
 50 |   ||| `fa`. If the evaluation of `fa` completes without encountering a cancelation, the finalizer
 51 |   ||| is unregistered before proceeding.
 52 |   ||| 
 53 |   ||| Note that if `fa` is uncancelable (e.g. created via `uncancelable`) then `fin` won't be
 54 |   ||| fired.
 55 |   ||| 
 56 |   ||| ```
 57 |   |||   onCancel (uncancelable(_ => canceled)) fin <-> F.unit
 58 |   ||| ```
 59 |   ||| 
 60 |   ||| During finalization, all actively registered finalizers are run exactly once. The order by
 61 |   ||| which finalizers are run is dictated by nesting: innermost finalizers are run before
 62 |   ||| outermost finalizers. For example, in the following program, the finalizer `f1` is run
 63 |   ||| before the finalizer `f2`:
 64 |   ||| 
 65 |   ||| ```
 66 |   |||   onCancel (onCancel canceled f1) f2
 67 |   ||| ```
 68 |   ||| 
 69 |   ||| In accordance with the type signatur, finalizers must not throw observable
 70 |   ||| errors
